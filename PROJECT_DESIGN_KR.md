## 1. 프로젝트 최종 목표

로컬 Ollama 모델 + CrewAI 멀티에이전트를 활용한 **웹소설 협업 집필 데스크탑 GUI 앱**.

- UI: CustomTkinter 기반 로컬 데스크탑 앱
- LLM: 로컬 Ollama (현재 gemma2:9b 기준, 추후 변경 가능)
- 배포: PyInstaller EXE
- 핵심 가치: 사용자가 방향을 잡고, AI 에이전트팀이 협업하여 집필

---

## 2. 전체 사용 흐름

```
[앱 실행]
    ↓
[프로젝트 선택 화면]
  ├─ 기존 프로젝트 선택 → 해당 작업 폴더 로드
  └─ 새 프로젝트 생성 → 작업 폴더 자동 생성 → 초기화 단계 진입
    ↓
[초기화 단계] ← 새 프로젝트에서만 진행
    ↓
[방 선택]
  ├─ 📝 소설 집필방
  ├─ ⚙️  설정 정리방
  └─ 🗂️  에피소드 관리방   ← 집필방과 완전히 분리된 별도 화면
```

## 3. 초기화 단계 (새 프로젝트 시작 시)

### 목적
작업 폴더 구조 생성 + 소설의 기반 문서(세계관, 줄거리, 설정 등) 작성

### 활성 에이전트
**편집자 에이전트 단독** 활성화
- 작가는 창작 욕구가 앞서 방향이 흐트러질 수 있음
- 편집자는 구조적 질문으로 설정의 빈틈/모순을 짚어내기에 적합
- 기록자는 문서 생성 시점에만 등장

### 흐름
```
[앱이 작업 폴더 자동 생성]
              ↓
[편집자] 사용자 아이디어 수신 및 조율 시작
  - 사용자가 아이디어 노트(txt/md)를 지정 폴더에 넣으면 앱이 읽어서 전달
  - 파일 크기 초과 시 자동 청크 분할 후 순차 처리 / 초과 경고 표시
  - 편집자가 구조적 질문으로 빈틈 메우기
              ↓
[사용자] [조율 완료] 버튼 클릭
              ↓
[편집자] 대화 내용 총정리 출력
              ↓
[기록자] 호출 → 초기 문서 일괄 생성
              ↓
[집필방 진입 가능 상태로 전환]
```

### 초기화 단계에서 생성되는 문서
- `settings/세계관.md` — 세계관 설정
- `settings/줄거리.md` — 전체 큰 그림/줄거리
- `settings/소설설정.md` — 인물, 능력, 규칙 등
- `context/story_context.md` — 초기 빈 파일 (집필 시작 후 기록자가 채움)
- `context/character_relations.md` — 초기 인물 인지 상태 (편집자 조율 내용 기반)

## 4. 에이전트 구성 및 역할

### ✍️ 작가 에이전트
- 활성 시점: 집필방
- 사용자의 레파토리 + 세계관 문서 + 맥락 요약을 바탕으로 소설 본문 집필
- 집필 완료 후 **구조화된 설계 의도**를 함께 출력 (기록자/편집자가 추론 없이 참조)
  ```
  확정된 사실:
  - 캐릭터 A가 B의 정체를 알게 됨
  - X 장소가 파괴됨

  심은 떡밥:
  - C의 과거 암시 (회수 예정 화 명시 가능)

  기존 설정 변경:
  - 없음
  ```

### 🔍 편집자 에이전트
- 활성 시점: 초기화 단계(단독) / 집필방 / 설정 정리방
- **초기화 단계**: 구조적 질문으로 세계관/설정 조율
- **집필방**: 본문 + 설계 의도 + `story_context.md` + `character_relations.md` 전부 대조, 혹독하게 검토
  - 떡밥이 갑자기 잊혀지진 않는지
  - 이야기 흐름이 튀거나 새진 않는지
  - 세계관 설정과 충돌은 없는지
  - 설계 의도가 본문에 실제로 구현됐는지
  - 캐릭터가 알 수 없는 정보를 언급하진 않는지
  - 독백/내면 묘사가 다른 캐릭터에게 인지되고 있진 않는지
- 검토 결과를 번호 목록으로 어필
  - 예) "✅ 3화 떡밥 A가 이번 화에서 회수됩니다."
  - 예) "⚠️ 이 부분은 세계관 설정 X와 충돌합니다."
  - 예) "⚠️ A는 아직 B의 능력을 알 수 없는 상황입니다."
- **설정 정리방**: 새 아이디어 vs 기존 설정 충돌 검토

### 📦 기록자 에이전트
- 활성 시점: 초기화 단계 완료 시 / 집필방 화 승인 후
- 세 가지 파일 관리 (모두 파일 1개씩 유지, append 없이 매번 전체 재작성)
- **저장 전 반드시 사용자 검수** (승인/거절. 거절 시 사용자가 직접 수정 후 저장)
- 할루시네이션 방지: 본문 직접 분석 없이, 작가의 구조화된 사실 목록만 참조

  **① 본편 파일** (`chapters/n화.txt`)
  - 확정된 소설 본문 저장 (화마다 새 파일 생성)

  **② 맥락 요약 파일** (`context/story_context.md`)
  - 파일 1개 유지, 매 화 완료 시 전체 재작성
  ```
  [떡밥 목록]
  - n화: 떡밥 내용 (미회수/회수)

  [이야기 흐름]
  - 1화: 핵심 사건 요약
  - 2화: ...

  [확정된 설정 변경]
  - 변경/추가된 세계관 설정
  ```

  **③ 인물 관계 및 인지 상태 파일** (`context/character_relations.md`)
  - 파일 1개 유지, 매 화 완료 시 전체 재작성
  ```
  [캐릭터 A] (n화 기준)
  ✅ 알고 있음: B의 이름 (1화), C가 조직 소속 (3화)
  ❌ 모르고 있음: B의 능력, B의 과거
  🤔 의심/추측 중: B가 뭔가 숨긴다는 느낌 (4화~)
  ```

## 5. 집필 파이프라인

```
[사용자] 이번 화 방향/아이디어/레파토리 입력
              ↓
[작가] 소설 본문 + 구조화된 설계 의도 출력 → 채팅창 표시
              ↓
[편집자] 본문 + 설계 의도 + 맥락 파일 전부 대조
        → 문제 목록 번호별로 어필 → 채팅창 표시
              ↓
[사용자] 항목별 판단 후 확정 지시 전달
    ✅ 이건 유지  🔧 이건 고쳐줘 + 내 의견
              ↓
    ├─ 수정 있음 → [작가] 확정 지시만 받고 재작성 (컨텍스트 리셋)
    │              → [편집자] 통과/불통과만 판정
    │              → 불통과 시 사용자에게 재판단 요청 (최대 2라운드)
    │
    └─ 전부 승인 → [기록자] 초안 출력 → 사용자 검수 → 저장 확정
```

### 루프 설계 원칙
- 매 라운드 컨텍스트 리셋: 작가는 항상 (설정 + 맥락요약 + 레파토리 + 최신초안 + 최신지시)만 받음
- 루프 상한 2라운드: 2회 후에도 불통과 시 사용자가 직접 개입
- 피드백 주체는 편집자 + 사용자: 편집자가 문제 발견, 사용자가 최종 방향 확정 후 작가에게 전달

## 6. 설정 정리방 파이프라인

```
[사용자] 아이디어 덩어리 입력 또는 파일 투입
              ↓
[1단계 - 압축]
  편집자 에이전트가 원문을 구조화된 설정 항목들로 분해
  (파일 투입 시 앱이 크기 체크 → 초과 시 청크 분할 후 순차 처리)
              ↓
[2단계 - 충돌 검토]
  정리된 항목 vs 기존 설정파일 대조
  충돌 발견 시 사용자에게 하나씩 제시:
  "⚠️ [새 아이디어] - 기존 설정 [X]와 충돌합니다.
   → [새 아이디어 채택] [기존 설정 유지] [직접 수정]"
              ↓
[사용자] 항목별 결정
              ↓
[3단계 - 설정파일 반영]
  결정된 내용을 기존 설정파일에 반영
```

## 7. UI 아키텍처

### 7-1. 전체 화면 구조

```
[프로젝트 선택 화면]
    ↓
[초기화 단계] (새 프로젝트만)
    ↓
┌──────────────────────────────────────────────┐
│  [방 선택]  📝 집필방   ⚙️ 설정 정리방         │
├───────────┬──────────────────┬───────────────┤
│  좌 사이드 │     채팅창        │  파일 관리     │
│  (설정관리)│   (메인 영역)     │  사이드바      │
└───────────┴──────────────────┴───────────────┘
```

### 7-2. 좌 사이드바 — 설정 관리
- 세계관/설정 문서 선택
- 에피소드 아크 목록 및 현재 진행 에피소드 (집필방만)

### 7-3. 메인 — 채팅창
- 에이전트 발화를 발화자별로 구분하여 표시
- 현재 진행 중인 에피소드/화 상단 표시 (집필방만)
- 사용자 입력창
- 상황별 액션 버튼 (파이프라인 단계에 따라 활성화):
  - 집필방: [승인] [수정 요청]
  - 설정 정리방: [채택] [유지] [직접 수정]
  - 기록자 검수: [저장 확정] [직접 수정 후 저장]
  - 초기화 단계: [조율 완료]

### 7-4. 우 사이드바 — 파일 관리
- 현재 프로젝트 폴더의 전체 파일 목록 표시
- 파일별 [보기] [편집] [삭제] 기능
- [+ 새 파일 생성] 버튼
- 설정파일 직접 수정 경로 역할도 겸함 (별도 설정 정리방 없이도 접근 가능)

## 9. 기술적 고려사항 및 안전장치

### gemma2:9b 한계 대응
- 에이전트에게 항상 전달하는 것: 설정파일(필수 구역) + story_context.md + character_relations.md
- 설정파일은 [필수 항목] / [상세 항목] 구역 분리 → 필수는 항상, 상세는 현재 에피소드 것만
- 소설 본문 전문은 에이전트에게 전달하지 않음 → story_context.md가 대체
- 맥락 파일은 append 없이 전체 재작성으로 크기 통제
- 작가가 설계 의도를 구조화된 목록으로 명시 → 기록자/편집자 추론 부담 최소화

### 할루시네이션 대응
- 기록자는 본문 직접 분석 없이 작가의 사실 목록만 참조
- 기록자 출력은 저장 전 사용자 검수 필수

### 초기화 오염 방지 (시나리오 2)
- [조율 완료] 버튼 클릭 시 편집자가 확정 내용 요약 출력
- 사용자가 요약 검토 후 [문서 생성 확정] 클릭해야만 기록자 호출
- 사용자가 한 번 더 확인하는 구조로 오염 방지

### 자동 백업 시스템 (시나리오 8)
- 트리거: 기록자 작업 완료 시 (화 저장, 맥락 파일 업데이트, 초기화 완료 등)
- 방식: Python `shutil.copytree()` 또는 `zipfile`로 프로젝트 폴더 전체 스냅샷 (AI 불필요)
- 저장 위치: `backup/001_20260316_143022/` 형식 (작업번호 + 타임스탬프)
- 파일 관리 사이드바에서 백업 목록 확인 + 특정 시점 롤백 가능

### 화 삭제 시 안전장치 (시나리오 5)
- chapters/ 내 파일 삭제 감지 시 경고 팝업
- 백업 시스템과 연동하여 해당 시점 이전 백업으로 context 파일 롤백 제안
  ```
  "n화 삭제 시 맥락 파일이 불일치합니다. 백업에서 복원하시겠습니까?"
  → [백업에서 복원] [그냥 삭제]
  ```

### 앱 강제 종료 대응 (시나리오 9)
- 에이전트 출력이 채팅창에 표시되는 즉시 `temp_draft.json`에 자동 저장
- 앱 재실행 시 temp 파일 감지 → "이전 작업을 이어서 하시겠습니까?" 제안

### Ollama 서버 다운 대응 (시나리오 10)
- 3회 재시도 로직 (exponential backoff)
- 3회 실패 시 "Ollama 서버를 확인해주세요" 안내
- 재연결 후 temp_draft에서 이어서 진행

### 편집자 체크리스트 (프롬프트)
```
집필방 검토 항목:
1. 사용자 레파토리가 본문에 실제로 반영됐는가? (시나리오 3)
2. 떡밥이 갑자기 잊혀지진 않는가?
3. 이야기 흐름이 튀거나 새진 않는가?
4. 세계관 설정과 충돌은 없는가?
5. 설계 의도가 본문에 구현됐는가?
6. 캐릭터가 알 수 없는 정보를 언급하진 않는가?
7. 독백/내면 묘사가 다른 캐릭터에게 인지되고 있진 않는가?
```

### 2라운드 실패 시 안내 (시나리오 4)
```
"2회 수정 후에도 [지적 항목]이 해결되지 않았습니다."
→ [해당 항목 무시하고 진행] [직접 수정하기]
```

### 맥락 파일 간 불일치 감지 (시나리오 6)
- 사용자가 맥락 파일 직접 수정 후 저장 시, 편집자에게 두 파일 교차 확인 요청 (1회 짧은 호출)

### 설정 정리방 연쇄 충돌 감지 (시나리오 7)
- 충돌 해결 후 변경된 설정 기준으로 충돌 검토 재실행
- 충돌 없을 때까지 루프 (최대 3회 제한)

### inbox 다중 파일 처리 순서 (시나리오 1)
- 파일명 또는 생성일 기준 정렬 후 순차 처리
- 처리 전 목록과 순서를 사용자에게 표시 후 확인

### 현재 코드의 주요 문제점 (개선 필요)
- 작가→편집자→기록자 단순 1회 순차 실행 (루프 없음)
- lore.txt 무한 append 누적
- langchain_community.llms.Ollama deprecated 사용
- Task 간 context 파라미터 누락

## 10. 에피소드 & 화 구조

- **에피소드** = 스토리 아크 묶음 (예: "주인공 죽다" 1~10화) — 사용자가 직접 정의
- **화** = 개별 집필 단위 — 에이전트가 화 단위로 집필

### 에피소드 관리방 레이아웃
집필방과 완전히 분리된 별도 화면. 기존 집필방 3패널 레이아웃을 건드리지 않음.

```
┌──────────────┬─────────────────────┬──────────────────┐
│ 에피소드 목록 │   선택된 에피소드    │  해당 에피소드의  │
│              │      상세 정보       │    화 목록       │
└──────────────┴─────────────────────┴──────────────────┘
```

**에피소드 목록 패널**
- 에피소드 제목, 상태, 스토리 내 순서 표시

**에피소드 상세 패널**
- 제목, 짧은 설명, 에피소드 목표, 핵심 사건(3~5개 bullet), 상태 표시
- 상태값: `planned` / `in_progress` / `completed` / `on_hold`

**화 목록 패널**
- 화 번호, 제목, 완료 여부 표시
- 화는 집필방에서 작성하는 실제 단위

### v1 에피소드 기능 범위
포함:
- 에피소드 생성 / 이름 변경 / 상태 변경 / 순서 조정 / 활성 에피소드 지정 / 설명·목표 편집

v1 제외 (과도한 복잡성):
- 칸반보드, 타임라인 에디터, 드래그&드롭 플롯 구조, 복잡한 스토리 플래너

## 11. 소프트웨어 아키텍처 레이어

### 전체 레이어 구조
```
GUI
 ↓
LoopController
 ↓
AgentFactory
 ↓
LLM Provider  (CrewAI 호환 LLM 객체 반환)
 ↓
CrewAI Agents
 ↓
Model (Ollama 등)
```

### LoopController
- GUI가 에이전트 루프 로직을 직접 갖지 않도록 분리
- 책임: 작가 호출 → 편집자 호출 → 수정 여부 판단 → 최대 2라운드 강제 → 결과 상태 반환
- v1: `ManualLoopController` (수동 while 루프)
- 미래: `CrewFlowLoopController` 로 교체 가능하도록 인터페이스 분리

```
LoopController (interface)
  ├─ ManualLoopController   ← v1 구현체
  └─ CrewFlowLoopController ← 추후 교체 가능
```

### AgentFactory
- LoopController로부터 컨텍스트를 받아 CrewAI 에이전트를 생성
- 주입 받는 것:
  - LLM Provider가 반환한 CrewAI 호환 LLM 객체
  - 현재 컨텍스트 파일 (settings, story_context.md, character_relations.md)
- 컨텍스트를 에이전트 backstory에 주입하여 생성
- 매 루프 라운드마다 새 에이전트 인스턴스 생성 (컨텍스트 리셋 원칙 유지)

### LLM Provider
- 역할: 모델 설정을 관리하고, CrewAI/LangChain이 수용하는 LLM 객체를 반환하는 팩토리
- v1: `OllamaProvider` → 내부적으로 `langchain_ollama.OllamaLLM(...)` 생성 후 반환
- 미래: `OpenAIProvider`, `AnthropicProvider` 추가 가능

```
BaseLLMProvider (interface)
  ├─ OllamaProvider   ← v1 구현체, OllamaLLM 반환
  ├─ OpenAIProvider   ← 추후
  └─ AnthropicProvider ← 추후
```

### 모델 선택 UI
- Ollama 설치 여부 확인: `ollama list` 실행 → 실패 시 에러 안내
- 설치된 모델 자동 감지: `ollama list` 파싱 → 드롭다운 목록 구성
- 모델 설정은 `project_config.json`에 저장

```json
{
  "llm_provider": "ollama",
  "llm_model": "gemma2:9b"
}
```
- 이전에 선택한 모델이 더 이상 존재하지 않으면 UI에 경고 표시
